### ADR-016: Langfuse Cloud, SDK v4, Manual Spans Over Framework Integration, No Circuit Breaker
Date: July 8, 2026 (Cloud-vs-self-hosted, v2 pin, no-breaker). Revised July 9, 2026 (v2 → v4,
masking, GroqInstrumentor) after installing Langfuse's official Claude Code skill and following
its documentation-first workflow.

Decision: Use Langfuse's free Cloud tier (API keys only, no infrastructure to run). Use the
current v4 SDK (not the v2 this ADR originally pinned). Instrument manually
(`start_as_current_observation` + `propagate_attributes`) rather than the LangChain/LangGraph
`CallbackHandler` framework integration. Add `GroqInstrumentor` for automatic model/token capture
on Groq calls. Do not add a circuit breaker around any of it.

---

Context — Cloud over self-hosted (unchanged from the original decision): the blueprint's original
wording said "self-hosted." Self-hosting Langfuse means an extra deployed service (its own
database, its own hosting bill, its own uptime to manage) for a portfolio project already running
on free tiers everywhere else (Render, Vercel, Supabase, Groq). The Cloud free tier gives the same
per-agent-step tracing with zero additional infrastructure.

Context — why v2 → v4 (this ADR's original reasoning was wrong on the numbers): the original
version of this ADR pinned v2 to avoid v3+'s OpenTelemetry dependency tree, citing "a real memory
cost on Render's 512MB free tier" without having actually measured it. Revisited this while
following the Langfuse skill's "always use the latest version" guidance. A real measurement
(installing both in clean venvs, importing each, comparing process RSS) found:

| Install | Measured import RSS delta |
|---|---|
| `langfuse==2.60.10` alone | ~27 MB |
| `langfuse==4.13.1` alone | ~31 MB |
| `langfuse==4.13.1` + `openinference-instrumentation-groq` | ~34 MB |
| `langfuse==4.13.1` + full `langchain` + `openinference-instrumentation-groq` | ~49 MB |

The marginal cost of v4 over v2 is ~4MB — both already pull in `pydantic`/`httpx` as shared
FastAPI dependencies, so the OTel packages are the only real addition, and they're small. That's
negligible next to `langgraph`'s own ~100MB already accepted on this same 512MB Render instance.
"Real RAM cost" was true in direction but wrong in magnitude; a measurement beats an assumption.
The 32-hex-trace-id constraint this ADR originally flagged as needing "a second id per session"
also turned out to be a non-issue: a UUID's 32 hex digits (dashes stripped) is exactly OTel's
trace-id format, so `research_sessions.id` converts directly with no second id anywhere. Pinned
`langfuse==4.13.1` (current latest as of this revision) and
`openinference-instrumentation-groq==0.1.16`.

Context — manual instrumentation over the LangChain/LangGraph `CallbackHandler` (this is the one
place this ADR deliberately goes against Langfuse's own general advice to "prefer framework
integrations over manual instrumentation"): the row above shows *why* — the `CallbackHandler`
needs the full `langchain` package, not just `langchain-core` (which `langgraph` already needs),
measured at ~15MB heavier than manual spans plus `GroqInstrumentor`. But the deciding factor isn't
the RAM difference, it's privacy. The `CallbackHandler` auto-captures each LangGraph node's *full*
input/output state as span data — for ARGUS that means every retrieved chunk's full content and
the full synthesized answer, sent to a third-party cloud service on every single query. That
directly contradicts ADR-013 (Groq and HuggingFace are the only disclosed sub-processors that see
full document content) and the Debug Diary's own existing rule that `execution_steps.detail` is a
truncated summary, never full content (BLUEPRINT.md line 517). Manual spans
(`backend/app/services/observability.py`'s `traced_span()`) send exactly what
`step_writer.py` already decided to send — the same short `trace_detail` string written to
Postgres — so Langfuse never sees more than the Postgres diary does. `mask_otel_spans` (truncates
any span attribute over 300 chars) is added as defense in depth on top of that, even though our
own spans are short by construction.

Context — `GroqInstrumentor` for model/token capture: Langfuse's own skill lists model name and
token usage as *baseline* trace requirements, not nice-to-haves — they're what makes a trace
dashboard show cost and enable model comparison. Manual spans don't capture these on their own.
Rather than hand-roll it, `GroqInstrumentor().instrument()` (`openinference-instrumentation-groq`)
patches the installed `groq` SDK process-wide, once, at startup — every existing `AsyncGroq` call
in `orchestrator.py`, `synthesizer.py`, and `critic.py` gets automatic model+token capture with
zero changes to those files. It must run *after* the Langfuse client is constructed (that's what
registers Langfuse as the active OpenTelemetry tracer provider — verified against the official
Groq+Langfuse cookbook, which instruments in this exact order) and its auto-created spans nest
correctly under whichever `traced_span()` block is open, because `start_as_current_observation`
sets the ambient OTel context for the duration of its `with` block and the Groq call happens
inside that block.

Context — no circuit breaker (unchanged conclusion, now on firmer ground): OpenTelemetry's
`BatchSpanProcessor` exports spans on a background thread; a failed or dropped export never blocks
or raises on the calling thread. This was Langfuse-specific reasoning under v2's own custom
batching; under v4 it's standard, well-documented OTel behavior, not a claim specific to this one
SDK. Verified directly, not just asserted: a smoke test constructed a real v4 client with
syntactically-valid but fake API keys, created spans, and let the background exporter attempt
delivery. The export failed with `401 Unauthorized` — Langfuse's own exporter printed it and moved
on. Nothing in this process caught fire or raised. `groq_breaker` and `hf_breaker` guard real
synchronous, blocking calls; Langfuse's emit path is neither, so a breaker here would still be
dead code pretending to protect something.

What actually protects the research flow: `traced_span()` is a context manager whose *setup* is
wrapped in try/except and falls back to a no-op span object on any failure (missing keys, bad
`session_id`, an SDK error) — the wrapped node then runs exactly as if tracing didn't exist.
Critically, once setup succeeds, exceptions raised by the *wrapped node itself* are allowed to
propagate normally; this context manager only guards its own setup/teardown, never the caller's
control flow, so a real node failure still reaches `step_writer.py`'s existing error-handling path
untouched. `mark_span()` (the post-execution `span.update()` call) is separately wrapped so a bad
`detail` value can't matter either. Missing `LANGFUSE_PUBLIC_KEY`/`LANGFUSE_SECRET_KEY` disables
the client cleanly with one startup log line.

Consequence:
- `backend/requirements.txt`: `langfuse==4.13.1`, `openinference-instrumentation-groq==0.1.16`.
- Root `.env.example`: `LANGFUSE_PUBLIC_KEY`/`SECRET_KEY`/`BASE_URL` (renamed from `LANGFUSE_HOST`
  — that's the v4 SDK's current env var name; anyone with the old key set on Render needs to
  rename it, same value).
- `backend/app/services/observability.py`: rewritten around `traced_span()` (a context manager,
  not a post-hoc emit call), `mark_span()`, `_mask_otel_spans()`, and `_instrument_groq()`.
- `backend/app/services/step_writer.py`: `traced()`'s wrapped node now runs *inside* the Langfuse
  span (`with traced_span(...): result = await node_fn(state)`) rather than being traced after
  the fact — required for Groq calls made inside a node to nest under that node's span.
- `/health/circuit-breakers` unchanged in shape: `langfuse` still reports an `{enabled, disabled}`
  flag, not a breaker state.
- `circuit_breaker.py`'s header comment already states Langfuse doesn't get one, and why.
