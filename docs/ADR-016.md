### ADR-016: Langfuse Cloud Over Self-Hosted, and No Circuit Breaker
Date: July 8, 2026
Status: Accepted

Decision: Use Langfuse's free Cloud tier (API keys only, no infrastructure to
run) instead of self-hosting, pin the v2 SDK (not v3), and do not add a
circuit breaker around it despite that being the original plan.

Context — Cloud over self-hosted: the blueprint's original wording said
"self-hosted." Self-hosting Langfuse means an extra deployed service (its own
database, its own hosting bill, its own uptime to manage) for a portfolio
project already running on free tiers everywhere else (Render, Vercel,
Supabase, Groq). The Cloud free tier gives the same per-agent-step tracing
with zero additional infrastructure. The tradeoff — data leaves ARGUS's own
infrastructure — is acceptable here since the traces contain latency/status
metadata, never raw chunk or answer content (same privacy stance as the
Debug Diary's `execution_steps.detail` column).

Context — v2 over v3: Langfuse's v3 SDK is built on OpenTelemetry, which pulls
in the OTel SDK's dependency tree — a real memory cost on Render's 512MB free
tier — and requires 32-hex-character trace ids, which would mean generating a
second id per session instead of reusing the existing `research_sessions.id`
UUID. v2 is a small batching client, has no OTel dependency, works with the
`httpx==0.27.0` already pinned for Supabase calls, and accepts the session
UUID directly via `trace(id=str(session_id))`. Pinned `langfuse==2.60.10`
(the newest 2.x release as of this ADR).

Context — no circuit breaker (deviates from the sketch in `PHASE3.md`'s
original Sprint 3a.4 plan, which named a `langfuse_breaker`): the Langfuse v2
SDK queues trace/span events onto an in-process background thread. The
`trace()`/`span()` calls invoked from `step_writer.py` never perform network
I/O on the request path, and delivery failures on that background thread
never propagate back to the caller. A circuit breaker exists to stop a caller
from repeatedly hitting a failing synchronous dependency — there is no such
call here to guard. Adding one anyway would be dead code that *looks* like
protection without providing any, which directly contradicts this project's
core rule that documentation and code claims must be honest about what
actually protects the system. `groq_breaker` and `hf_breaker` guard real
synchronous, blocking calls; Langfuse's emit path is neither.

What actually protects the research flow instead: `observability.py`'s
`record_step_span()` is synchronous, wraps its own body in try/except, and
never raises — the same never-crash contract every other diary write in this
project follows (`record_step` in `step_writer.py`, the `research_sessions`
insert/patch calls in `main.py`). If Langfuse's client fails to init (bad
keys, network unreachable at startup) or an emit call throws for any reason,
it's caught, printed, and ignored. Missing `LANGFUSE_PUBLIC_KEY`/
`LANGFUSE_SECRET_KEY` disables the client cleanly with one startup log line —
research is unaffected either way, verified by design and confirmed by the
Langfuse-down live test (`docs/PHASE3-TEST-SCRIPT.md`, TC-3a.4-01).

Consequence: `backend/requirements.txt` (`langfuse==2.60.10`), root
`.env.example` (`LANGFUSE_PUBLIC_KEY`/`SECRET_KEY`/`HOST`, all optional),
`backend/app/services/observability.py` (new), a two-line hook in
`step_writer.py`, and `/health/circuit-breakers` reporting `langfuse` as an
`{enabled, disabled}` flag rather than a breaker state. `circuit_breaker.py`'s
header comment updated to state plainly that Langfuse does not get one, and
why.
