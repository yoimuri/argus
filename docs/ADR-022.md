# ADR-022: Report Generation (Sprint 4.6a) — engine, model split, async execution, metering

**Status:** Accepted, 2026-07-13
**Context:** Sprint 4.6a (plan decision D17). Report generation is the product's headline
capability: turn a collection of messy PDFs into a structured, formatted, downloadable report.
The Phase 4 plan required this sprint to confront two cautions head-on instead of inheriting the
Q&A pipeline's defaults, and to treat metering and honesty as part of the design.

---

## 1. The engine: whole-collection, not RAG — single-pass when it fits, map-reduce when it doesn't

The Q&A pipeline retrieves the top-5 most similar chunks — right for "answer a question," wrong
for "synthesize these documents into a report." A report built from five arbitrary passages would
be a summary of whatever the embedding search happened to like. The report engine therefore
bypasses `retriever.py` entirely and reads the whole collection.

**Revision 2 (2026-07-13, same day — the constraint that owns every number).** Two versions of
this engine failed live before the design finally started from the right constraint. Version 1
(sequential map-reduce, ~25 calls of ~5k tokens each) and version 2 (a 30k-token single pass +
concurrent map batches) were both sized against the model's **context window** (131k tokens). The
binding constraint is Groq's free-tier **rate meter**, verified against
console.groq.com/docs/rate-limits on 2026-07-13: **8,000 tokens/minute, 30 requests/minute,
200k tokens/day — per model**, input + output both counted. Against that meter, version 2's
single-pass call could *never* succeed (30k > 8k), version 1's second call in any minute 429'd,
and concurrency just turned pacing into a 429 storm. Worse, those failures counted against the
**shared** `groq` breaker, so a failed report run opened the breaker and degraded interactive Q&A
(orchestrator/synthesizer/critic all fail open) for the next minute — the live "slow Ask, low
confidence, then fine again" pattern.

The engine is now sized from the meter, not the context window:

- **Single-pass:** if the whole collection is ≤ `SINGLE_PASS_CHARS` (10k chars — content + prompt
  overhead + the 3,072-token output budget ≈ 6.1k tokens, one minute-window with headroom), ALL of
  it goes into one reduce call. Higher quality than map-reduce (no lossy intermediate step) and
  ~1 minute total.
- **Map-reduce (larger collections):** 18k-char batches (~5.7k tokens/call incl. output) condensed
  into dense notes **sequentially** — under a shared per-minute meter, concurrency cannot speed
  anything up — then one reduce call over the notes (input head-trimmed to
  `MAX_REDUCE_INPUT_CHARS` so it also fits one window).
- **Client-side token budgeter** (`_pace_for_tokens`): every report Groq call declares its
  estimated spend (chars/4 + max_tokens) against a 6,500/min budget (headroom under the true 8k)
  and sleeps until the window rolls if it doesn't fit. Pacing up front beats discovering the meter
  via 429s and the SDK's slow hidden retries (which is what "took forever and then failed" was).

**Two isolation guarantees, so a report can never break Q&A again:**
1. **Quota isolation:** ALL report calls (classify, template distill, map, reduce) run on
   `openai/gpt-oss-120b` — Groq meters each model separately, and the Q&A pipeline runs entirely
   on `gpt-oss-20b`, so a report run consumes zero of the interactive budget. (Quality is equal or
   better; the map notes now come from the larger model. Both env-overridable:
   `GROQ_REPORT_MODEL`, `GROQ_REPORT_MAP_MODEL`.)
2. **Failure isolation:** report calls go through their own `groq_report` breaker
   (circuit_breaker.py's own rule: two unrelated failure modes must not share one circuit).
   Visible as its own card on the SOC panel via `/status/breakers`.

**Honest bounds:** map work is capped (4 batches/document, 8 map calls/run ⇒ a worst-case run is
~8 paced minutes, inside the 20-minute stale window). A collection past the cap is *sampled evenly
across its length* rather than truncated at the front, the notes are labeled "evenly sampled, not
exhaustive," and the reduce prompt sees that label. The free-tier meter makes an unbounded run
impossible to promise; sampling honestly beats reading 40% and implying 100%. Known residual: the
D18 helper's truncation retry re-runs a call without re-pacing it — rare (low-effort calls with
explicit structure seldom truncate), and the budgeter's headroom absorbs it.

Model IDs verified against Groq's live model list on 2026-07-13 (the Llama 3.x models are
deprecation-listed; Llama 4 Scout is preview-only and reserved for 4.6c's vision work). All calls
go through `call_reasoning_json()` (ADR-018/D18), so the reasoning-token truncation trap stays
handled in one place.

## 2. The process lesson (recorded so it isn't repeated)

The original model-choice question ("which model is best?") was answered by verifying model IDs
and context windows — but never the **quotas**. On a free tier, capacity design must start from
the provider's rate meter (TPM/RPM/TPD), because the meter, not the model, decides what a call can
look like. This is now part of the project's verify-external-facts rule: checking that a model
exists is not checking that your calls to it are possible.

## 3. Templates: encoded report expertise, three trust tiers

A report's *shape* is half its value (the same collection should always produce a
professionally-structured document, where a free chat gives a different shape each ask):

1. **Built-in** — cybersecurity and data-science section structures ship in code.
2. **Web-looked-up** — for any other recognized domain, a Tavily search for that report type's
   conventional structure, distilled to a section list by the small model. Every snippet passes
   the same `injection_patterns.matches_any` scan as Web Scout's results **before any model reads
   it** (quarantined snippets are logged to `security_events` as `web_content_as_instruction`,
   source `report_template_lookup:<url>`). Web text advising a template is exactly as untrusted
   as web text answering a question.
3. **General** — a sound generic structure whenever classification or lookup fails. Every failure
   in the template path degrades here; none of them can block the report.

Document text itself is untrusted input throughout: chunks were regex-scanned at upload
(quarantined ones never reached the database), and every prompt in this pipeline carries the same
data-not-instructions framing the synthesizer uses. Same structural ceiling as ADR-007/ADR-012 —
more coverage, not a removed ceiling.

## 3b. Report from a completed session (added 2026-07-13)

A report can also be generated from a completed research session instead of a whole collection
(`POST /reports` takes exactly one of `collection_id` or `session_id`). The session's
already-synthesized answer becomes the reduce's source material directly — classify + one reduce
call, no map, no re-reading documents. The point is cost/latency reuse: a user who already asked a
question and got a good answer can turn *that* into a formatted, downloadable report without paying
to re-process every document. It is still metered as one report (`usage_events`), because it still
runs the large model once. The report row's `collection_id` is set to the session's collection
(SET NULL-safe), and the reduce is framed by the original question so it keeps that focus.

## 4. Asynchronous execution: the row is the interface

A single-pass run is one large model call (seconds on a warm dyno); a map-reduce run over a large
collection is several. Either way, holding the HTTP request open for the duration is unwise: the
Sprint 4.3 cancel rework proved twice, live, that Render's proxy buffers request cycles
unpredictably, so a request that runs longer than the proxy's tolerance can be cut without the
backend ever seeing a clean abort.

So `POST /reports` creates the `reports` row (`status='running'`, migration 017) and returns
immediately; the generation runs in an in-process `asyncio` task; the frontend polls
`GET /reports/{id}` until the row reaches a terminal state. Polling also keeps the free-tier dyno
awake for the duration of the run. Cancel is the established DB-signal idiom:
`POST /reports/{id}/cancel` flips the row, the generator checks the flag between model calls, and
the completed-write is filtered `&status=eq.running` so a finished report can never overwrite a
cancellation.

**Honest limits, stated plainly:**
- This is an in-process task, **not a job queue**. A dyno restart mid-generation orphans the run.
  Mitigation: `GET /reports/{id}` marks a `running` row older than 20 minutes as `error`, so the
  UI always reaches a terminal state. The real fix is the long-deferred async-jobs
  rearchitecture; this ADR does not claim to be it.
- A failed run records **why** in `reports.error_detail` (migration 018, revision 2) — a short
  user-safe sentence (rate limit hit / daily quota spent / service paused / interrupted run) shown
  on the report page, so a failure is diagnosable instead of a bare "Generation failed". Both
  writes fall back to a detail-less patch if migration 018 isn't applied yet (deploy-order
  safety, same lesson as migration 010).
- The task authenticates with the user's JWT captured at request time. Tokens outlive any
  realistic run length (~1h vs. minutes), but a run started seconds before expiry could fail its
  final write — it would then land in the stale-row path above.

## 5. The deliverable: .docx + print-CSS PDF, disclaimer non-optional

- **`.docx`** via `python-docx` (1.2.0, the one approved new dependency — small, pure-Python,
  imported lazily by the download endpoint only). Built on demand from the stored Markdown;
  nothing binary is persisted. The converter handles exactly the Markdown the reduce prompt is
  allowed to emit (headings, lists, bold) — a constrained generator paired with a simple
  converter, not a Markdown engine.
- **PDF** via print-CSS on the preview page (`window.print()`), with a print media block that
  forces the light token set so dark-theme users don't print white-on-white. Server-side PDF
  (weasyprint-class) stays deferred: heavy system deps on a 512 MB dyno.
- **The disclaimer is part of the design** (owner decision, 2026-07-11): "AI-generated draft —
  proofread before use" renders in the preview banner, is embedded under the .docx title, and is
  deliberately *not* print-hidden. The tool makes the work easier; it does not replace the human
  pass.

## 6. Metering: usage_events, exactly as migration 014 anticipated

Report generation is the costliest flow in the app (a couple of calls for a single-pass or
session-based run; up to ~11 paced calls plus a possible Tavily search for a large map-reduce
run). The daily cap counts `usage_events` rows (`event_type='report'`) — append-only,
no client delete, no collection FK — against `usage_limits.max_reports_per_day` (migration 017;
tight default 3, existing accounts backfilled to the QA tier). Deleting a report, or its source
collection, never refunds a unit: same bypass-proof reasoning as the research cap, decided before
launch rather than after a live bypass this time.

## 7. Explicitly out of scope for 4.6a

Figure generation (4.6b — charts from document data, needs a real memory test of
matplotlib-vs-SVG on the 512 MB dyno) and multimodal image reading (4.6c — vision captioning,
which gets its own threat-model ADR first; image-borne injection is a new channel). The landing
page continues not to claim either until they ship.
