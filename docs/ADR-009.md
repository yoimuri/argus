### ADR-009: 30-Minute Idle Timeout, Not an Absolute Session Cutoff
Date: July 5, 2026
Decision: Sessions expire after 30 minutes of no requests, tracked via an httpOnly
`last_active` cookie checked and refreshed in `proxy.ts` on every request. This is
not an absolute cutoff, active use resets the timer indefinitely.
Context: Supabase's default session refresh already keeps an active user logged in
indefinitely, `proxy.ts` refreshes the token on every request. An absolute 30-minute
cutoff was considered and rejected: this app gets demoed live to recruiters, an
absolute timer risks logging out mid-demo regardless of activity, a real cost with
no matching security benefit for a portfolio-stage app. Idle timeout gives the
actual security property (an unattended, logged-in session doesn't stay open
forever) without that risk.
Consequence: `supabase.auth.signOut()` is called with `scope: 'local'`, not the
default `scope: 'global'`. Global would end every session for that user on every
device, not just the idle one, confirmed against Supabase's own docs before writing
this, the default would have been a real, easy-to-miss bug if shipped without
checking. Enforcement happens lazily, on whatever the next request happens to be,
not a live countdown, a tab left open with zero activity doesn't get auto-redirected
until something triggers a new request against it.
Status: Accepted. Not part of Phase 2's sprint structure, same reasoning as the CSP
headers and CORS fix, this is session/auth hardening, a different domain than Phase
2's LLM/agent security scope, fixed in passing rather than given its own sprint.

Addendum, July 6, 2026: user testing found the idle timeout never actually fired.
Two real bugs, both in `proxy.ts`:
(1) The `last_active` cookie was written with no `maxAge`, making it a session
cookie — the browser deleted it on tab/window close. So "leave the site, come back
later" always looked like a fresh visit with no prior activity timestamp, and the
countdown never triggered. Exactly the scenario this ADR exists to cover (an
unattended session should expire). Fix: persist `last_active` with an explicit
maxAge so it survives the browser being closed, which is what makes a real away-
and-return actually enforce re-login. The timer still resets on every request, so
active use never gets logged out mid-demo (the original ADR-009 intent is intact).
(2) On the idle-expiry path, `signOut()` cleared the auth cookies by mutating
`supabaseResponse`, but the function then returned a *different* NextResponse.redirect
object that never received those cleared-cookie headers — so the browser kept its
session and the "logout" was cosmetic. Fix: copy the cookie mutations onto the
redirect response before returning it, and explicitly delete `last_active`.
Lesson, same family as the rest of this project: a security control that is written
but silently no-ops is worse than none, because it reads as covered. Only found by
actually testing the away-and-return path, not by reading the code.
