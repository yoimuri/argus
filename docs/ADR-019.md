# ADR-019 — Public surface, Google sign-in, and usage limits (Sprint 4.4)

**Status:** Accepted (code-complete 2026-07-11; not live-verified). Supersedes the "Not built"
rows in ADR-013's pre-launch checklist for the specific items this sprint touches, and updates the
rest honestly rather than silently.

## Context

Through Phase 3 and Sprints 4.1–4.3, ARGUS was effectively a private tool: `/` force-redirected to
`/dashboard`, so the only way in was a login wall, and the only accounts were the owner's test
users. Phase 4's premise is making the system *visible*. Sprint 4.4 takes the first real step
across the public boundary:

1. `/` becomes a public marketing/landing page (a recruiter following the repo link should see the
   project, not a login form).
2. Google (Gmail) OAuth is added as a signup path, per the locked plan decision — Gmail-first
   signup with tight, user-visible free-tier limits.
3. A public signup surface that runs billable LLM/embedding calls per request cannot be
   unmetered — the owner pays Groq and HuggingFace quota.

Crossing toward a public surface re-raises everything ADR-013 parked under "pre-launch checklist."
This ADR records what was built, and — the honest part — what building it does and does not
discharge.

## Decision 1 — Public landing at `/`, exposing no user data

`/` is a static marketing page (server component + a small client reveal island). It renders the
project pitch, the real agent pipeline, the real security story, and contact links. It reads the
caller's auth state only to switch a button between "Sign in" and "Go to dashboard" — it exposes no
document, session, or security-event data to anyone. `proxy.ts` gains a centralized `isPublicPath()`
that matches `/` **exactly** (never as a prefix) plus the already-public `/login` and `/auth/*`.
Every `/dashboard/*` route keeps its dual guard (proxy redirect + per-page `getUser()`).

**Why safe to ship publicly now:** a marketing page with no data exposure carries none of ADR-013's
data-protection obligations. The public boundary that *does* carry them is signup (Decision 2), not
the landing page.

## Decision 2 — Google OAuth mechanism built; enabling real public signups stays gated

The OAuth flow is `@supabase/ssr`'s default PKCE: `signInWithOAuth` → Google → `/auth/callback`
route handler → `exchangeCodeForSession` → session cookies. The callback is hardened (open-redirect
guard on `?next=`, Vercel `x-forwarded-host` handling) and stamps `last_active` on success so an
OAuth login shares the password path's double-login immunity.

**What this does NOT discharge (the honesty point):** building the button is not the same as being
ready for real external users. The moment real people can sign up, ARGUS is processing their data
through third-party sub-processors (Groq, HuggingFace) with no privacy policy and no disclosure.
Those are ADR-013's items, and they are *still not written*. So:

- The **mechanism** is code-complete and can be enabled for the owner's own testing.
- **Opening it to the public** remains a deliberate, gated decision — it should not happen until the
  checklist below has real answers. Enabling the Google provider in Supabase is Clint's manual
  action precisely so this stays a conscious choice, not an accident of deploying code.

This is the same discipline as the rest of the project: the capability existing in the repo must not
be read as the capability being safely live.

## Decision 3 — Usage limits: metered, un-forgeable, owner-editable

`usage_limits` (migration 011) holds one row per user: `max_collections`, `max_documents`,
`max_research_per_day`. Enforced in the backend (`create_collection`, `upload_document`, `research`)
with a friendly 429 before any billable work runs.

**Trust model — clients cannot raise their own caps.** `usage_limits` grants `SELECT` only to
`authenticated`, with an own-row RLS policy; there is no insert/update/delete grant and no write
policy. A user reads their caps (for the dashboard meter) but can never change them from the browser
or a forged request. Only the `SECURITY DEFINER` signup trigger (which sets tight defaults for new
accounts) and the owner (via Supabase Studio / service role) can write them. A missing row fails
**closed** to the tight defaults — never "unlimited", never a 500. New signups get 3 / 15 / 15;
existing accounts were backfilled to an owner/QA tier so current testing isn't blocked.

## Consequences

- Recruiters can now reach a real landing page; the repo link no longer dead-ends at a login form.
- The owner can test Google sign-in end to end once the provider is configured.
- Free-tier quota is protected the instant signups are enabled — metering is in place first, not
  bolted on after a bill.
- The legal/privacy work is *named and tracked*, not quietly assumed done.

| Gap | Status after Sprint 4.4 |
|---|---|
| Google OAuth sign-in | **Built (code), 🟡 not live-verified.** Mechanism ready; enabling for the public still gated on the rows below. |
| Usage/quota limits on a public surface | **Built (code), 🟡 not live-verified** (this ADR / migration 011). Not an ADR-013 row originally, but a real prerequisite for any public signup — done first, on purpose. |

| Sub-processor disclosure (Groq, HuggingFace named) | **Still not written.** |
| Verify Groq/HF data retention & training-use terms | **Still not reviewed.** |
| Data retention policy (inactive-data expiry) | **Still not decided** (currently: kept indefinitely; session auto-expiry parked in BACKLOG — needs `pg_cron`). |
| Data export (GDPR Art. 15/20) | **Still not built.** |
| Account-level "delete all my data" | **Still not built** (per-collection erasure exists — ADR-013's delete endpoint). |

The bottom four-plus rows are the real gate on flipping Google sign-in on for the world. This ADR
does not close them and does not pretend to.
