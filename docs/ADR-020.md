# ADR-020 — Account deletion with a grace period (and what "deleted" honestly means)

**Status:** Accepted (code-complete 2026-07-11, 🟡 not live-verified). Extends ADR-013's
right-to-erasure from per-collection to account level. Gate: GATE-26.

## Context

ADR-013 shipped per-collection erasure and explicitly listed "account-level delete all my data" as
not built. Clint requested it 2026-07-11 with a specific shape: a red, GitHub-style danger-zone
button, a type-`DELETE` confirmation, and a **7-day grace period** — the account is scheduled for
deletion, the user can withdraw within the window, and after it expires the deletion is final and
unrecoverable.

## Decision — two-phase deletion, enforced on next contact

**Phase 1, request (client-side, RLS-scoped):** Settings → Danger zone (a GitHub-style row: label +
description, outlined red button) → clicking "Delete account" pops out a confirmation dialog — the
typed `DELETE` confirmation is never shown inline by default, only after the explicit click
(corrected live 2026-07-11: the first version showed the input inline, which skipped the
GitHub-standard "click to reveal the point of no return" gate) → typing `DELETE` and confirming
stamps the user's own `user_profiles.deletion_requested_at` (migration 015). Nothing else changes:
the account works normally through the grace period, every dashboard page shows a banner with the
deletion date and a withdraw link, and withdrawing simply nulls the column. Both writes go through
the user's own token under the existing "own profile" RLS policy — no new privileges.

**Phase 2, finalize (backend, on first visit after expiry):** the dashboard layout reads the
profile on every page render. When `now > deletion_requested_at + 7 days`, `DeletionNotice`
finalizes before the user can do anything else: calls `DELETE /account` (backend), which purges —
with the user's own token, under RLS — every collection (Storage files, documents, chunks), every
research session (execution_steps cascade), then stamps `account_deleted_at`; the client then signs
out and lands on `/login?reason=deleted` ("This account was deleted... can't be restored"). Any
later sign-in attempt hits the layout, sees `account_deleted_at`, and is signed straight back out.

**Why "next contact" instead of an exact-time job:** this stack has no scheduler (no pg_cron, no
worker — the same honest gap that parked session auto-expiry in BACKLOG). The grace period is
enforced *correctly* (nothing can be done with an expired account; the first expired visit purges
before rendering anything), just not *proactively* (an expired account nobody ever touches keeps
its data until touched). When pg_cron or a worker lands, a daily sweep completes this.

## What "deleted" covers — and the one thing it doesn't

**Purged permanently:** all collections, all documents + their Storage PDFs, all embedded chunks,
all research sessions + execution traces, and the account is locked out of the app.

**Deliberately NOT purged:**
- `usage_events` — metering must not be user-erasable (migration 014's entire point; deleting your
  account on day 6 must not reset a rate limit for a re-signup).
- `security_events` — the defensive audit trail (consistent with the SOC posture).

**Known limitation, stated plainly:** the Supabase Auth identity itself (`auth.users` row — email,
hashed credentials) is NOT removed, because deleting an auth user requires the service-role key,
and this backend deliberately holds only user-scoped tokens (a core security posture choice,
documented since the Phase 4/4b split). The account is *unusable* and its *data is gone*; the
identity row lingers until the Phase 4b service-key work (which BLUEPRINT already schedules for the
security_events write path) adds true identity removal — or the owner deletes the user by hand in
Supabase Studio (Authentication → Users) when asked. GDPR-grade full erasure = data purge (done) +
identity removal (owner-manual today, 4b-automatic later). The Settings copy and ADR-013's
checklist reflect exactly this, no more.

## Alternatives rejected

- **Immediate hard delete:** no recovery from a rash click or a hijacked session; the grace period
  is the industry-standard mitigation (GitHub, Google) and was explicitly requested.
- **Service-role key on the backend now:** would enable full identity deletion but breaks the
  user-token-only trust model everywhere else for one feature; 4b introduces it deliberately with
  its own review, not as a side effect.
- **Purge at request time, lock for 7 days:** withdrawal would be meaningless (data already gone).
  The point of grace is that withdrawal restores *everything*.
