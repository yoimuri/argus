### ADR-018: Shared Reasoning-JSON Helper, HF Embedding Breaker, and the Phase 4 / 4b Split
Date: July 9, 2026
Status: Accepted

This ADR bundles three Sprint 4.1 decisions that share one theme: closing gaps
that were found, worked around locally in each affected file, and never given
one authoritative fix or one honest doc home.

---

## Part 1 тАФ `call_reasoning_json()`: ending the recurring token-truncation bug

**Decision:** Add one function, `call_reasoning_json()` in
`backend/app/services/llm_json.py`, that owns every Groq reasoning-model call
across the three agents that make one (`orchestrator.py`, `synthesizer.py`,
`critic.py`). It reads `finish_reason` after the call, retries once at a
lower `reasoning_effort` if the response truncated (`finish_reason ==
"length"`), and raises a typed `ReasoningTruncated` if the retry also
truncates. Each agent keeps its own fail-open `except` block unchanged тАФ
`ReasoningTruncated` is just one more exception that lands in it.

**Context тАФ why this needed a dedicated fix, not another comment block.**
`openai/gpt-oss-20b` is a reasoning model: it spends hidden reasoning tokens
before any visible content, and Groq's `max_tokens` caps the two combined. A
long reasoning pass can silently starve the visible output, and the SDK
raises no exception when that happens тАФ the only signal is
`finish_reason == "length"` on an otherwise-200 response. This exact failure
mode has now hit the project three separate times, in three different
agents, each discovered live rather than in review:

1. **Sprint 3a.1 (2026-07-07/08).** The original vague-query bug ("summarize
   for me" returning "no relevant information found") was first misdiagnosed
   as a weak classifier, then as a retriever bug тАФ both wrong. Opus traced it
   with live evidence (reproducing the real Groq call locally) to reasoning
   tokens swinging 348тАУ547+ on an identical prompt, occasionally exhausting
   the budget. Fixed in `synthesizer.py` (`reasoning_effort: "low"`) and
   `orchestrator.py` (`max_tokens` 300тЖТ512, later 768). Full trail in
   `CONTINUITY.md`'s 2026-07-07 entries; `docs/ADR-014.md` covers the
   Orchestrator's sub-query design from the same sprint but does not itself
   document the token-budget fix тАФ that lived only in code comments and
   `CONTINUITY.md` until now.
2. **Sprint 3b (2026-07-09).** Adding the `use_web` field grew the
   Orchestrator's system prompt by a full paragraph, which measurably
   increased how often reasoning ran long enough to truncate the JSON. Live
   testing caught it directly (2 of 4 `execution_steps` traces showed
   `JSONDecodeError` on truncated content). Fixed same day: `max_tokens`
   768тЖТ1024. Full account in `docs/ADR-017.md`'s revision section.
3. **The Critic (`critic.py`)** independently landed on `max_tokens=1536` and
   `reasoning_effort="medium"` for the same reason, worked out by inference
   from the other two agents' comments rather than from a shared source of
   truth.

Each occurrence produced its own hand-tuned `max_tokens`/`reasoning_effort`
pair and its own paragraph of comment explaining the trap тАФ but none of the
three ever read `finish_reason`, the one field that would have told them
definitively "this call truncated" instead of leaving `extract_json()` to
fail on a cut-off JSON string and get lumped in with every other kind of
parse failure. The bug kept recurring because the fix each time was local
knowledge (a comment, a CONTINUITY.md entry) rather than a shared mechanism тАФ
found via the `context-engineering` skill trial on 2026-07-09, which flagged
that three agents independently carrying near-identical unread comments
about the same trap is itself the signal that the trap needs one place to
live, not a fourth copy of the same paragraph.

**Design.**
- Both the initial attempt and the lower-effort retry happen **inside a
  single `breaker.call()`** тАФ an actual Groq outage (an exception on either
  attempt) counts as one breaker failure, not two, mirroring the same
  principle applied to the HF embedding retry in Part 2 below. Two separate
  `breaker.call()` wraps around each attempt would double-count every real
  outage and open the breaker at half the intended threshold.
- Truncation itself does **not** count as a breaker failure тАФ it's a content
  problem, not an availability problem, and the pre-helper code never fed it
  to the breaker either (the truncation-driven `extract_json()` failure
  always happened *outside* `groq_breaker.call()`, in the agent's own `try`
  block). The helper preserves that: `ReasoningTruncated` is only raised
  after `breaker.call()` returns normally.
- Effort steps down one rung on truncation (`highтЖТmediumтЖТlow`, floored at
  `low`, since gpt-oss-20b has no effort below that тАФ a `low`-effort call
  that still truncates gets retried at `low` again, which is a second roll of
  the dice, not a guaranteed fix, since reasoning length varies run-to-run at
  a fixed effort).
- **No behavior change on the happy path.** Every agent's existing
  `max_tokens` and `reasoning_effort` values carried over unchanged
  (orchestrator: 1024/medium, synthesizer: 1024/low, critic: 1536/medium) тАФ
  this sprint only adds the truncation check and retry, it does not re-tune
  any budget.

Consequence: `backend/app/services/llm_json.py` gains `call_reasoning_json()`
and `ReasoningTruncated`; `orchestrator.py`, `synthesizer.py`, `critic.py`
migrate their Groq calls onto it, and each file's duplicated token-trap
comment block collapses into a one-line pointer to this ADR. No migration, no
new env var.

**What this does not fix.** A genuinely long/complex query can still exhaust
even the retry тАФ the helper makes that failure detectable and gives it one
extra chance, it does not remove the underlying reasoning-model constraint
that hidden and visible tokens share one budget. Any agent that needs a
larger structural budget still has to raise its own `max_tokens`, same as
before; the helper only stops that need from hiding behind a silent parse
failure.

## Part 2 тАФ `hf_embedding_breaker`: closing BACKLOG Item 6

**Decision:** Wrap `document_processor.py`'s HuggingFace feature-extraction
calls in a new, dedicated `hf_embedding_breaker` (`circuit_breaker.py`), with
one retry nested *inside* the breaker call, and detect HF's cold-start
`{"error": "...model loading..."}` 200-body on the batch/chunk embedding path
(`embed_chunks`), which тАФ until this sprint тАФ had no response validation at
all, unlike `embed_query`.

**Why a new breaker, not reusing `hf_breaker`.** `hf_breaker` already guards
the prompt-injection classifier call in `injection_guard.py`, which treats a
breaker-open state as "skip the AI classifier layer, fall back to
regex-only" тАФ a safe degradation for the security guard specifically.
Sharing one breaker between the classifier and the embedding endpoint would
mean an embedding-service outage (which has nothing to do with the injection
classifier's health) could trip the same circuit and silently downgrade
security detection as a side effect. Two unrelated failure modes, two
breakers тАФ this is the same reasoning ADR-012 already applied when the
classifier got its own breaker separate from `groq_breaker`.

**Retry-inside-breaker, not breaker-around-retry.** `_hf_embedding_with_retry()`
does attempt тЖТ one retry тЖТ raise, and *that whole function* is the single
unit passed to `hf_embedding_breaker.call(...)`. A real HF outage therefore
counts as one breaker failure per request, not two. Wiring it the other way
(a breaker wrap around each individual attempt) would double-count every
real outage and reach `fail_threshold` at half the number of actual failed
requests тАФ an easy inversion to get backwards, so it's called out explicitly
here and in the code.

**The `embed_chunks` gap.** Before this sprint, `embed_query` validated its
HF response (rejecting anything that wasn't a real 384-float vector,
including an error dict), but `embed_chunks` тАФ used by the upload path's
batch embedding тАФ returned the raw HF response with zero validation. A
cold-start `{"error": "..."}` dict would pass straight into
`iter_embedded_chunk_batches()`'s `zip(batch, embeddings)`, which would
iterate the dict's *keys* as if they were 384-float embedding vectors,
producing a confusing, hard-to-diagnose upload failure instead of a clear
one. Fixed by moving the error-dict check into the shared
`_hf_embedding_once()` that both `embed_query` and `embed_chunks` route
through, so the check now applies once, to both callers, instead of only the
one that happened to have it.

**Also while touching this code:** the per-attempt HTTP timeout dropped from
60s to 30s, so a worst-case attempt-then-retry sequence tops out around ~63s
instead of ~123s inside a synchronous `/research` or upload request on
Render's free tier, rather than doubling the old already-long timeout.

**Failure surfaced to the caller.** `CircuitBreakerOpen` (already carrying a
`.retry_in_s` estimate) now bubbles out of both the upload handler and
`/research` as a clean `503` with a retry hint, instead of the previous
uncaught-exception path, which would have surfaced as a generic `500` in
`/research` (the retriever's `embed_query` call has no local try/except) and
was already caught-and-500'd in the upload handler's broad `except Exception`
(now specifically caught first and turned into a `503`). `/health/circuit-
breakers` gains an `hf_embedding` entry alongside `groq`, `hf_prompt_guard`,
and `tavily`.

Consequence: `circuit_breaker.py` (+`hf_embedding_breaker`),
`document_processor.py` (`_hf_embedding_once`/`_hf_embedding_with_retry`/
`_call_hf_embedding`, 30s timeout), `main.py` (`CircuitBreakerOpen` handling
in upload + `/research`, `hf_embedding` health entry). No migration, no new
env var. Closes `docs/BACKLOG.md` Item 6.

## Part 3 тАФ Phase 4 / 4b split

**Decision:** Phase 4, as actually built starting this sprint, is a
**read-only dashboard with no role system**: per-user breaker health, a
per-user security-events feed (Realtime, RLS-scoped), a research-session
history with an execution-timeline view, a cleaned-up report UI, a public
landing page, and Google sign-in with owner-editable usage limits. Everything
in `docs/BLUEPRINT.md`'s original Phase 4 sketch that assumes an admin
role or tables that don't exist тАФ world map, IP intelligence, a global
request feed, `admin_settings`, `request_log`, `circuit_breaker_log`,
`/soc/*` and `/admin/*` routes, service-key `security_events` writes, an
AI session summarizer тАФ moves to an explicitly named **Phase 4b**, not built,
not implied as built.

**Why this is a Part of ADR-018 and not just a ROADMAP edit.** BLUEPRINT.md
is written in the voice of a pre-build specification and has stayed largely
unedited on this point since June 2026; nothing in the repo previously said
outright that the admin-role SOC vision was out of scope for the phase
titled "Phase 4." Leaving that unstated risks exactly the failure this
project's own rule exists to prevent (docs describing a more complete system
than the code has) the moment Phase 4's dashboard ships and a reader compares
it against BLUEPRINT's original SOC section. Writing the split down here,
alongside the code that draws the actual line, is the same discipline
`docs/ADR-013.md` applied to the privacy/E2E-encryption question.

**What's buildable now with existing RLS, no schema work beyond this
sprint's migration 009:** every user already has row-level access to their
own `security_events` and `research_sessions`; Realtime honors that same
policy per-subscriber. A per-user events feed and a per-user session history
need no admin concept at all. The breaker snapshot endpoint
(`/health/circuit-breakers`) is process memory, not a table, and was already
built in Phase 2/3.

**What's blocked on 4b specifically:** anything that needs to see *other
users'* data (a global request feed, a world map of all traffic) requires
either a real admin role with its own RLS policies or a service-key write/read
path that bypasses RLS entirely тАФ both are real design decisions (who can
grant admin, how is it revoked, what does the service key path look like
given `security_events` is currently written with the *caller's own* token,
a caveat migration 004 already flags) that deserve their own planning pass,
not a rider on Phase 4's dashboard work.

Consequence: `docs/BLUEPRINT.md`'s Phase 4 roadmap section, SOC description,
database schema table, and API surface all get corrected in this same
sprint's doc pass (see the Sprint 4.1 entry in `docs/PHASE4.md` and
`docs/ROADMAP.md` for the itemized before/after). No code consequence beyond
what Parts 1тАУ2 already describe тАФ this Part is a documentation-honesty fix,
not a technical decision with its own diff.
