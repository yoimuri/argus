### ADR-008: Static CSP Over Nonce-Based CSP for the Frontend
Date: July 5, 2026
Decision: Security headers (CSP, X-Content-Type-Options, X-Frame-Options,
Referrer-Policy, Permissions-Policy) are set via `next.config.ts`'s `headers()`
function, using a static Content-Security-Policy rather than a nonce-based one.
Context: A third-party pentest scanner flagged all four as missing on the deployed
Vercel frontend. CSP is the highest-signal finding: it connects directly to
LLM05:2025 (Improper Output Handling), currently marked closed in BLUEPRINT.md's
compliance table on the strength of "markdown-only output, no executable paths"
alone, a code-level claim. CSP is the browser-level backstop for the same risk.
Next.js's own documentation is explicit that nonce-based CSP requires generating a
fresh nonce per request in proxy.ts, forces every page into dynamic rendering, and
disables static optimization and ISR. None of ARGUS's pages currently rely on static
optimization in a way that would be lost, but the nonce approach adds real setup and
testing surface for a portfolio-stage project. The static route needs
'unsafe-inline' on script-src and style-src, Next.js's default hydration and
Tailwind's inline styles need it without a nonce. This is a real, known reduction
in what CSP protects against, an inline-script-injection XSS would not be blocked
by this policy. It still blocks any externally-sourced script, restricts frame
embedding via frame-ancestors, and restricts object-src and base-uri.
Consequence: Meaningfully better than no CSP, not the strongest possible CSP.
connect-src is scoped explicitly to the Supabase project and the Render backend, not
left as a wildcard. Upgrading to a nonce-based policy is a legitimate future
refinement if a stricter script-src is ever needed, not required at this stage.
Status: Accepted. Not part of Phase 2's sprint structure, that phase is scoped to
LLM/agent security on the backend. This is browser-level hardening on the frontend,
a different domain, fixed in passing rather than given its own sprint, same
treatment as the CORS wildcard fix.

Addendum, July 5, 2026: the first deployed version of this CSP hardcoded
`argus-27g9.onrender.com` as the allowed backend origin, sourced directly from
PHASE1-HANDOFF.md's documented Live URLs table. That URL was already stale, the
Render backend had been migrated to a different account (the original account's
free-tier quota was being consumed by an unrelated n8n project), which assigns a
new random subdomain, `argus-am5t.onrender.com`, on any account move. CSP correctly
blocked the mismatch, that's the control working as intended, but the reference
value it was checking against was wrong. Lesson: an external URL hardcoded into a
security allowlist is a live fact, not a fixed one, verify it's still current
before trusting a doc, the same discipline this project already applies to package
versions and platform defaults.

Addendum, July 7, 2026 — reversed: moved to nonce-based CSP after all. A
third-party pentest scanner flagged `script-src 'unsafe-inline'` as HIGH severity
(7.5, finding HDR_003): it disables XSS protection outright, any injected
`<script>` or inline event handler executes unrestricted. The original tradeoff
this ADR made (avoid nonce-based CSP's forced dynamic rendering) turned out to
cost less than assumed for this specific app: ARGUS's pages are already
auth-gated and already read cookies per request, so the "forces dynamic
rendering" cost was mostly being paid already, not a new cost being taken on.
For a security-focused portfolio project, shipping a scanner-confirmed HIGH
finding on the live demo also outweighs the setup cost this ADR originally
avoided.

Implementation: `next.config.ts` no longer sets `Content-Security-Policy` (a
per-request nonce can't be a static header); `proxy.ts` now generates
`Buffer.from(crypto.randomUUID()).toString('base64')` per request and builds
`script-src 'self' 'nonce-{value}' 'strict-dynamic'` (no `unsafe-inline`).
`style-src 'self' 'unsafe-inline'` is kept deliberately: components use inline
`style={{}}` throughout, nonces don't apply to style *attributes* at all (only
`<style>`/`<script>` tags), and style injection can't execute JS — nonce-ing it
would cost real UI work for no security gain. `connect-src` now reads
`process.env.NEXT_PUBLIC_SUPABASE_URL`/`NEXT_PUBLIC_API_URL` instead of the
hardcoded values this ADR originally shipped, closing the exact stale-hardcoded-URL
failure mode the addendum above already found once.

Real bug found during verification, not assumed to work from the docs: Next's
own documentation states nonces are auto-applied to inline hydration scripts,
but that only holds for *dynamically rendered* pages. `/login` built as a
statically-generated page (Next marked it `○ Static`), and curling it directly
confirmed the problem concretely — its inline `self.__next_f.push(...)` hydration
scripts shipped with no `nonce` attribute at all, which a strict `script-src`
with no `unsafe-inline` fallback would block outright, breaking hydration
in production. Adding `export const dynamic = 'force-dynamic'` directly to
`login/page.tsx` had **no effect** (confirmed: build still showed `○ Static`,
served bytes identical before/after) because that file had `'use client'` at
its top — route segment config exports are only recognized on Server Component
modules, and a client-boundary page silently drops them, no error or warning.
Fix: split the page the same way `dashboard/page.tsx` already does — a thin
Server Component `login/page.tsx` (carries `export const dynamic =
'force-dynamic'`) rendering a new client component `login/LoginForm.tsx` (the
actual interactive form, previously the whole file). Re-verified after the
split: `/login` now builds as `ƒ Dynamic`, and two separate curls to the live
route show the nonce differs per request and matches between the response
header and the HTML body's `nonce="..."` attributes on every script tag.

Consequence: `script-src` no longer contains `unsafe-inline`, closing HDR_003.
`style-src 'unsafe-inline'` remains, a deliberate, documented residual gap
(style-only injection, not script execution). The lesson from the `/login`
bug generalizes beyond this one page: any future statically-renderable page
under this CSP needs the same Server-Component-wrapper treatment, not a bare
`export const dynamic` on a `'use client'` page.
Status: Accepted, supersedes the original decision above (kept for the
history, not deleted, same practice as every other ADR reversal in this repo).
