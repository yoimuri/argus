### ADR-008: Static CSP Over Nonce-Based CSP for the Frontend
Date: July 5, 2026
Decision: Security headers (CSP, X-Content-Type-Options, X-Frame-Options,
Referrer-Policy, Permissions-Policy) are set via `next.config.ts`'s `headers()`
function, using a static Content-Security-Policy rather than a nonce-based one.
Context: A third-party pentest scanner flagged all four as missing on the deployed
Vercel frontend. CSP is the highest-signal finding: it connects directly to
LLM05:2025 (Improper Output Handling), currently marked closed in BLUEPRINT.md's
compliance table on the strength of "markdown-only output, no executable paths"
alone, a code-level claim. CSP is the browser-level backstop for the same risk.
Next.js's own documentation is explicit that nonce-based CSP requires generating a
fresh nonce per request in proxy.ts, forces every page into dynamic rendering, and
disables static optimization and ISR. None of ARGUS's pages currently rely on static
optimization in a way that would be lost, but the nonce approach adds real setup and
testing surface for a portfolio-stage project. The static route needs
'unsafe-inline' on script-src and style-src, Next.js's default hydration and
Tailwind's inline styles need it without a nonce. This is a real, known reduction
in what CSP protects against, an inline-script-injection XSS would not be blocked
by this policy. It still blocks any externally-sourced script, restricts frame
embedding via frame-ancestors, and restricts object-src and base-uri.
Consequence: Meaningfully better than no CSP, not the strongest possible CSP.
connect-src is scoped explicitly to the Supabase project and the Render backend, not
left as a wildcard. Upgrading to a nonce-based policy is a legitimate future
refinement if a stricter script-src is ever needed, not required at this stage.
Status: Accepted. Not part of Phase 2's sprint structure, that phase is scoped to
LLM/agent security on the backend. This is browser-level hardening on the frontend,
a different domain, fixed in passing rather than given its own sprint, same
treatment as the CORS wildcard fix.

Addendum, July 5, 2026: the first deployed version of this CSP hardcoded
`argus-27g9.onrender.com` as the allowed backend origin, sourced directly from
PHASE1-HANDOFF.md's documented Live URLs table. That URL was already stale, the
Render backend had been migrated to a different account (the original account's
free-tier quota was being consumed by an unrelated n8n project), which assigns a
new random subdomain, `argus-am5t.onrender.com`, on any account move. CSP correctly
blocked the mismatch, that's the control working as intended, but the reference
value it was checking against was wrong. Lesson: an external URL hardcoded into a
security allowlist is a live fact, not a fixed one, verify it's still current
before trusting a doc, the same discipline this project already applies to package
versions and platform defaults.
