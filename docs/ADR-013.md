### ADR-013: Data Privacy & Compliance Posture
Date: July 8, 2026
Status: Accepted

Decision: Document ARGUS's actual data-privacy posture honestly. What protects
user data today, what does not, and what a real public launch would still
require, rather than letting "we have encryption" stand in for a real answer.
Implement user-initiated deletion (`DELETE /collections/{id}`) now as the one
concrete gap that was actually missing. Do not implement auto-delete-on-logout.

Context: A direct question was asked. Is user data safe, should it be
encrypted, and are we compliant enough to let real users try this if Google
sign-in is added later? The honest answer required separating three things that
get casually conflated: encryption, isolation, and erasure.

**What already protects data:**
- **In transit:** TLS everywhere (Vercel, Render, Supabase all serve HTTPS).
- **At rest:** Supabase encrypts its underlying storage by default (platform-level
  AES-256). This is real and already covers ARGUS.
- **User isolation:** Postgres Row Level Security (`001_core_schema.sql`) means
  one user's rows are invisible to another user's queries, enforced at the
  database, not just in application code. Also real and already done.

**What encryption does NOT do, stated plainly because this is where people get
it wrong:** none of the above hides document content from ARGUS's own backend
or operator. A database transparently decrypts rows for the query that's
authorized to read them. At-rest encryption protects against someone stealing
the raw disk, not against the application (or whoever runs it) reading the
data it's designed to process. The only thing that would hide content from the
operator is application-level end-to-end encryption, where the server only
ever holds ciphertext.

**Why E2E encryption is not on the table for this specific app, not a gap to
"fix" later:** ARGUS's entire function is reading document text to chunk it,
embed it, and let an LLM answer questions about it. A RAG pipeline cannot
operate on ciphertext. Chunking, embedding, and synthesis all require the
plaintext. "We can never see your documents" and "we answer questions about
your documents" cannot both be true for this architecture. Any documentation
claiming otherwise would repeat the exact doc-vs-code overclaiming problem this
project's core rule exists to prevent. This is a structural property of RAG
generally, not an ARGUS shortcoming.

**The actual levers for a real privacy posture are policy and control, not
crypto:**
1. RLS. Done (above).
2. **Right to erasure (GDPR Art. 17)**. A user must be able to delete their own
   data on demand. This did not exist: there was no delete endpoint at all.
   Implemented now: `DELETE /collections/{id}` removes the collection row
   (cascading to its documents and document_chunks per the schema's `on delete
   cascade`) and purges the underlying PDF files from Supabase Storage
   (`storage_path`, added in migration 005, since the upload path was
   previously discarded and never persisted. See that migration's own
   comment). This is real erasure, not a partial one that leaves orphaned
   files sitting in Storage.
3. **Third-party sub-processor disclosure**. Document text is sent to two
   external services for processing: **Groq** (LLM inference. Sees full
   chunk content and the user's query) and **HuggingFace** (embeddings and
   the prompt-injection classifier. Sees chunk content and query text).
   Both are real sub-processors under any standard privacy-policy framework.
   Their retention/training terms have not been reviewed for this ADR. That
   review, plus disclosing these processors to users, is a pre-launch
   requirement, not something already satisfied.

**Rejected: auto-delete on logout or on starting a new collection.** This was
the original framing of the request that prompted this ADR. Rejected because:
it destroys the user's own document library on every session boundary (directly
contradicts the ask to be able to reopen previous collections), and it would
make Phase 3/4's planned session-history features (research_sessions,
execution_steps/Debug Diary, SOC dashboard) pointless. There would be nothing
persistent for them to show. A user who wants their data gone now has the
correct tool: delete it themselves.

Consequence: For the current stage (one test user, pre-launch), data persisting
as-is with RLS + platform encryption is a reasonable, honestly-stated posture.
The pre-launch checklist below is explicitly **not** built by this ADR. It is
the list of what "ready for real users" actually requires, so it isn't lost or
silently assumed done:

| Gap | Why it matters | Status |
|---|---|---|
| Privacy policy (public-facing document) | Legal requirement before any real signups | Not written |
| Sub-processor disclosure (Groq, HuggingFace named) | Users have a right to know who processes their data | Not written |
| Verify Groq/HF data retention & training-use terms | Determines what disclosure needs to say | Not reviewed |
| Data retention policy (how long inactive data is kept) | Currently: forever, no expiry | Not decided |
| Data export (GDPR Art. 15/20: access & portability) | Users can request their own data back | Not built |
| Account-level "delete all my data" (not just per-collection) | Full account erasure, not just document-by-document | Not built |
| Google OAuth sign-in | Mentioned as a future auth option | Not built: still email/password only |

Status: Accepted. The delete endpoint above is built and is a genuine
compliance improvement; everything in the checklist table is deliberately
unbuilt and should stay that way until this project is actually preparing for
real external users, not claimed as done in the meantime.
