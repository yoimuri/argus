### ADR-015: Bounded Re-Retrieval Loop + Critic
Date: July 8, 2026. Revised July 8, 2026 (refusal-badge backstop added).
Status: Accepted

Decision: Add a Critic agent after the Synthesizer that grades the draft
answer against its retrieved chunks and, only when it can name what's
missing, triggers exactly one re-retrieval pass targeting that gap. Hard cap
the loop at 2 total passes (OWASP ASI10 — no self-modifying, unbounded agent
loops).

Context: Nothing validated a synthesized answer against its sources before
this. A confident-sounding but unsupported answer looked identical to a
well-grounded one. The Critic closes that gap and gives the report a visible
confidence signal instead of a silent one.

**Retry is meaningful, not automatic.** Re-running the retriever with the
exact same `refined_queries` returns the exact same chunks — a retry that
doesn't change what's searched for cannot change the outcome, so looping on
low confidence alone would just burn a Groq call and latency for nothing. The
Critic's JSON output includes an optional `retry_queries` field: 1-2 short,
concrete queries naming what the draft was missing. `needs_retry` is only
`True` when confidence is low **and** at least one of those queries is novel
(not already in `refined_queries`). Those queries get appended (capped at 5
total sub-queries) for the retry pass, so the second search actually looks
somewhere new.

**Where `loop_count` lives.** It's incremented inside `critic_node` itself
(`state.get("loop_count", 0) + 1`) rather than by the graph's conditional
edge. That means it's already part of state by the time the router evaluates
`needs_retry and loop_count < 2` — a missed seed value can't produce an
uncapped loop, because the increment doesn't depend on prior seeding to have
happened correctly. LangGraph's default `recursion_limit` (25) is the
backstop fence behind that.

**Fail-open, same stance as the Orchestrator.** Any failure — Groq down,
breaker open, malformed JSON, missing `flags` — returns empty
`confidence_flags` and `needs_retry=False`. A broken Critic must never block
the report. The reporter renders "Not assessed" for empty flags rather than
defaulting to a "High" badge, since a successful Critic run always returns at
least one flag; empty flags unambiguously means the check didn't happen; a
default "High" here would overclaim.

**The diary showing agents twice on a retry is intended, not a bug.** A
retried run's `execution_steps` trace shows retriever, synthesizer, and
critic each appearing twice with a continuous `step_index` (0-7 total across
orchestrator through reporter) — that's the visible record of the self-check
loop firing, exactly the kind of thing the Debug Diary exists to show.

Consequence: `backend/app/agents/critic.py` (new), `backend/app/agents/
reporter.py` (confidence badge, pure string formatting, no LLM call), `state.py`
(`confidence_flags`, `needs_retry`, `loop_count`), `graph.py`'s first cycle
(`synthesizer → critic → {retriever | reporter}`), and `main.py` deriving
`research_sessions.status = 'completed_with_fallback'` from `loop_count >= 2`
after `ainvoke` returns rather than adding a separate `status` state field.
No migration — `research_sessions.status` is unconstrained text (migration
008).

**Revision, July 8, 2026 — deterministic refusal backstop for the badge.**
Live testing surfaced a real gap: the Critic's own system prompt says a
refusal draft ("the context doesn't say") should always grade low, but that's
an LLM judgment call, not a guarantee. The identical question, re-run,
produced both outcomes — low-then-recovers-to-high on one run, high
immediately on another — with the badge disagreeing about the same class of
answer across runs. Same shape of problem the injection guard already solved
(`injection_patterns.py` backs its LLM classifier with regex) and the same
fix: `critic.py`'s `REFUSAL_PATTERNS` checks the draft's own wording (its
"I'm sorry, but..." opener has been a reliable tell in practice) and, if the
model's flags say a refusal is fully grounded, appends an ungrounded flag so
the badge can't show High confidence on an answer that admits it couldn't
answer the question. Deliberately scoped to the badge only — it does not
force `needs_retry`, because a real retry needs the model's own gap-targeted
`retry_queries` to search for, which a regex can't produce; forcing a retry
with no new query to run would just repeat the same search and change
nothing (the same "retry must be meaningful" principle already established
above). No keyword list is complete against every refusal phrasing — same
honest limitation already accepted for the injection guard (ADR-007) — this
narrows the gap, it doesn't close it perfectly.
