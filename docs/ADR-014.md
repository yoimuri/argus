### ADR-014: Orchestrator Intent Routing via Sub-Query Expansion + the Vector-Index Fix
Date: July 8, 2026
Status: Accepted

Decision: Add an Orchestrator agent that classifies a query's intent
(`specific`/`broad`/`meta`) and rewrites it into concrete sub-queries before
retrieval, instead of adding a similarity-threshold argument to the
`match_document_chunks` RPC. Retrieval runs once per sub-query, merges and
dedupes the results by chunk id, and keeps the top-N by similarity.

Context: A vague query like "summarize this for me" returned "no relevant
information found." The naive fix would have been an RPC-level similarity
threshold or a hardcoded fallback for empty results. Both patch the symptom at
the database boundary without addressing the real problem: a generic query
embeds to a generic vector with no reliable target chunk. The chosen fix
teaches the *retrieval strategy* to sample broadly when the question itself is
broad, which is a general capability, not a special case for one query shape.

Fail-open stance: if Groq is down, the breaker is open, or the response is
unparseable JSON, the Orchestrator falls back to `intent='specific'`,
`refined_queries=[original query]` — exactly pre-Phase-3 behavior. The
Orchestrator improving retrieval must never be able to block retrieval.

**Revision during live testing (2026-07-08):** sub-query fan-out alone did not
fix the vague-query case. Retriever instrumentation (added to trace the
failure) showed the same collection returning 0, 1, and 1 rows for three
different sub-query embeddings — inconsistent with a working ANN index. The
root cause was the `ivfflat` index on `document_chunks.embedding`
(`lists = 100`, migration 001). `ivfflat` buckets vectors into 100 lists and,
at the default `probes = 1`, scans only the single nearest list. `lists = 100`
is tuned for a large dataset (rule of thumb ~rows/1000); on ARGUS's small test
collections the chunks are spread thinly enough that most lists are empty, so
a query vector landing on an empty list returns zero rows regardless of how
good the sub-query is. Different sub-query embeddings probe different lists,
which produced the random-looking 0/1/1 pattern. Fix: migration
`007_hnsw_vector_index.sql` replaces `ivfflat` with `HNSW`, which has
near-perfect recall by default with no tuning knob to get wrong.

This folds two decisions into one ADR because they're one retrieval-quality
story: sub-query expansion only works once the ANN index actually returns the
nearest chunks for each sub-query. Shipping the Orchestrator without the index
fix would have looked broken for reasons that had nothing to do with the
Orchestrator's own logic.

Consequence: `backend/app/agents/orchestrator.py` (new), changes to
`retriever.py` (fan-out + merge + dedupe), `graph.py` (new first node),
`state.py` (`intent`, `refined_queries`), and migration 007. Adding a
similarity-threshold RPC argument stays explicitly out of scope — noted as
deferred, not built, since the sub-query merge plus the index fix already
close the acceptance criterion without a schema change.
